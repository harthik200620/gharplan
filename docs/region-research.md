## CAD rendering + AI tech survey

### Recommended stack (2026)

Plain inline **SVG**, hand-rolled in **React/Next.js**, with a thin custom pan/zoom + editing layer. **No rendering library.** Keep one optional geometry helper only if you need true wall-offset math.

- **Renderer:** a single `<svg>` driven by React from a plain plan model (rooms as polygons in real-world mm/inches). Marked `"use client"`.
- **Pan/zoom:** one CSS-style transform (`scale` + `translate`) applied to a top-level `<g>`; you own the wheel/drag math (~40 lines). No zoom library required; if you want momentum/pinch ergonomics, `@use-gesture/react` (~10 KB, headless) is the only optional add.
- **Editing:** drag/resize handled by pointer events you attach yourself, hit-testing in world coordinates. Handles render in a screen-space overlay so they stay a constant pixel size at any zoom.
- **Geometry (OPTIONAL):** only pull in a library if you need boolean/offset operations â€” `polygon-clipping` (~10 KB, robust, headless) for room-from-centerline wall offsets and zone unions. Prefer this over maker.js/paper.js, which carry a full graphics runtime you do not need when React already owns the DOM.
- **DXF preview (OPTIONAL, only if importing/previewing real CAD):** `dxf-viewer` (vagran) â€” WebGL/Three-based, built for large real-world files, actively maintained. Mount it in its own `ssr:false` dynamic component, completely separate from the SVG editor. Do NOT use three-dxf (gdsestimating), which is ~4 years stale.

**Why SVG over Canvas/Konva/Fabric for this specific app:**
1. This is a low-element-count, text-and-line-heavy, label-rich drawing (dozens-to-low-hundreds of nodes, not 5k+). SVG's bottleneck (~3â€“5k DOM elements) is far above a residential plan; below that threshold SVG is competitive or faster than canvas, and far simpler.
2. Clean vector export is a hard requirement here. SVG IS the export: serialize the DOM to an .svg file, or print-to-PDF with crisp vectors and selectable text. Canvas/Konva export is rasterized (toDataURL) unless you maintain a parallel vector path â€” a large hidden cost.
3. Accessibility, CSS theming (dark mode via CSS variables), `<title>`/`<desc>`, and hit-testing come free from the DOM. Canvas reimplements all of it.
4. React's declarative model maps 1:1 onto SVG elements â€” your plan model renders directly to JSX, with React reconciling diffs. With Konva you adopt a second scene graph (react-konva) and lose direct DOM/CSS/print integration; with fabric.js you also fight the lack of official React bindings.

**When the recommendation flips:** choose react-konva/Konva if the plan must show thousands of simultaneous elements (large multi-floor commercial plans, dense furniture/MEP layers, live dragging of 1000s of nodes) or heavy raster compositing â€” that is canvas territory and Konva gives the best React-integrated canvas scene graph, transformer handles, and pinch-zoom out of the box. For a single residential floor plan, that performance headroom is unused and you pay for it in export quality and complexity. fabric.js and paper.js are not recommended for a React app: fabric has no official React bindings and overlaps Konva; paper.js is a self-contained vector/scene-graph runtime that duplicates SVG+React. maker.js is a geometry/DXF-generation library, not a renderer.

### Rendering approach

SVG, rendered declaratively by React. Single `<svg>` viewBox in world units (e.g. millimetres). One `<g transform="translate(tx,ty) scale(s)">` wraps all drawing layers â€” pan/zoom mutate only tx/ty/s, never the geometry. Apply `vector-effect="non-scaling-stroke"` to walls/dimension lines so line weights stay crisp (e.g. exterior 3â€“4px, interior 2px, door/window 1â€“1.5px) regardless of zoom â€” **the single most important trick for a CAD look.** Zone fills use SVG `<pattern>` for hatching and flat fills with low opacity for tints. The north arrow, scale bar, title block, toolbar, and edit handles render OUTSIDE the world transform in screen-space (fixed pixels) so they never zoom. Export = serialize the SVG to a string (Blob download) for vector .svg, and rely on the same SVG for print-to-PDF; rasterize via an offscreen `<canvas>` + `drawImage` only when a PNG is explicitly wanted. Mark the renderer `"use client"`; SVG itself SSRs fine, but the interaction layer needs the client. (Pure inline SVG has no Next.js SSR hazard â€” unlike react-konva, which must be loaded via `dynamic(..., { ssr:false })` because Konva touches the canvas/DOM at import time and 404s under Next 14 SSR.)

### Component architecture

**Layered renderer + screen-space overlay.**

**Data model (single source of truth, plain serializable objects in world units):**
- `Room { id, polygon: [[x,y]...], name, zone, areaCached }`
- `Wall { id, a, b, thickness, kind: 'exterior'|'interior' }` (or store wall rects directly)
- `Opening { id, wallId, offset, width, type: 'door'|'window', swing: 'L'|'R'|'in'|'out' }`
- `ViewState { scale, tx, ty }` and `EditState { selectedId, dragMode, hover }`

Hold it in Zustand (or useReducer/Context). Keep ViewState/EditState separate from geometry so panning/selecting never reflows the model and undo/redo only snapshots geometry.

**Component tree:**
```
<FloorPlanCanvas>                     // "use client"; owns wheel/pointer handlers; sets up <svg viewBox>
  <svg>
    <defs>                            // hatch <pattern>s, arrowhead/tick <marker>s, swing clip paths
    <g transform={world}>             // THE ONE pan/zoom transform â€” everything below is in world units
      <ZonesLayer/>                   // 1 drawn first: fills + hatching (tints behind walls)
      <WallsLayer/>                   // 2 thick strokes / offset polygons; non-scaling-stroke
      <OpeningsLayer/>                // 3 door arcs (swing), window double-lines, wall gaps
      <DimensionsLayer/>              // 4 witness + dimension lines, ticks, measured text
      <AnnotationsLayer/>             // 5 drawn last: room labels, computed areas
    </g>
    <ScreenLayer>                     // OUTSIDE the transform â†’ fixed pixel size at any zoom
      <NorthArrow/> <ScaleBar/>
      <TitleBlock/>                   // project / scale / date / sheet no.
      <EditOverlay/>                  // drag + resize handles, snap guides, marquee
      <Toolbar/>                      // mode toggles, zoom %
    </ScreenLayer>
  </svg>
</FloorPlanCanvas>
```

**Key decisions:**
- Layer = z-order. Each `*Layer` is a stateless component mapping a model slice to SVG. Render order in JSX is paint order. Splitting layers keeps re-renders local (memoize each on its slice).
- One transform, geometry untouched. Pan/zoom only edit `ViewState.{scale,tx,ty}`. To zoom-to-cursor: convert pointer screenâ†’world, scale, then adjust tx/ty so the world point stays under the cursor. ~40 lines, or use @use-gesture/react.
- Coordinate conversions: screenâ†’world = `((px - tx)/s)`. All hit-testing runs in world space; point-in-polygon for room selection.
- Editing overlay is screen-space. Handles are drawn in pixels (so they don't shrink when you zoom out) but positioned by projecting world corner points to screen. Dragging a handle updates the room polygon in world units; snapping rounds to a grid or to nearby wall lines.
- Crisp line weights: `vector-effect="non-scaling-stroke"` on walls and dimension lines so a 2px wall stays 2px at every zoom â€” the core CAD-look detail.
- Openings as parametric cuts: a door is a gap in the wall plus an arc `<path>` for the swing; a window is a gap plus a double line. Compute from `Opening.{offset,width,swing}` so editing a wall updates them.
- Export: an `<ExportButton>` serializes the live `<svg>` (XMLSerializer â†’ Blob) for vector .svg; the same markup gives crisp print-to-PDF. PNG only on demand via offscreen canvas. Because there is no second (canvas) scene graph, what you see is exactly what exports.
- DXF preview, if needed, is a SEPARATE island: `<DxfPreview>` using dxf-viewer behind `dynamic(ssr:false)` â€” never entangled with the editor's SVG.
- Next.js: the editor file is `"use client"`. Inline SVG SSRs without issue, so no dynamic-import dance is needed for the editor itself (only the optional dxf-viewer/Konva islands need ssr:false).

### Export approach

**Vector-first**, because the chosen renderer IS vector.
1. **SVG download:** `new XMLSerializer().serializeToString(svgEl)` â†’ `Blob(type 'image/svg+xml')` â†’ object URL â†’ anchor download. Inline computed styles or ship a `<style>` block in `<defs>` so the file is self-contained; embed fonts or stick to web-safe families for portable text.
2. **PDF:** rely on the SVG for browser print-to-PDF (a dedicated print stylesheet sizes the sheet and shows the title block); for server/programmatic PDF, feed the SVG to a vector-aware lib (e.g. svg2pdf.js + jsPDF, or render the route to PDF). Vectors stay sharp and text stays selectable.
3. **PNG (only when explicitly requested):** draw the serialized SVG into an offscreen `<canvas>` via an Image and `drawImage`, then `toBlob`.

The decisive contrast: with Konva/fabric/canvas, (1) and (2) require maintaining a parallel vector representation; with inline SVG the on-screen DOM is the export artifact, so there is zero divergence between preview and output.

### Anthropic structured-JSON recommendation

For a Next.js route handler returning a floor-plan room program as structured JSON:

**Tool/function calling vs JSON mode â€” the landscape changed.** "JSON mode" was never a real Anthropic switch; it was prompting workarounds (instruct-for-JSON, prefill `{`, or a dummy tool you force-call to harvest schema-shaped input). As of 2025/2026 those are obsolete. Anthropic now ships first-class **structured outputs** (GA on Claude Opus 4.8 / Sonnet 4.6 / Haiku 4.5 / Fable 5) with two independent features:
1. **JSON outputs** â€” `output_config: { format: { type: "json_schema", schema: {...} } }` constrains the whole response to your schema via grammar-constrained sampling. **This is what you want for a room program** (the model is producing data, not deciding whether to act).
2. **Strict tool use** â€” `strict: true` on a tool guarantees its `input` validates against the schema. Use this when the room program is one capability among several the model may call; force it with `tool_choice: { type: "tool", name: "..." }`.

For "the endpoint's job is to return one JSON object," prefer **(1) JSON outputs**. Reach for (2) strict tool use only when JSON generation is mixed into an agent that also calls other tools. Both beat the legacy prefill/dummy-tool hacks (which now 400 on these models â€” last-assistant-turn prefills are rejected).

**Recommended implementation (TypeScript route handler, `app/api/floorplan/route.ts`):**
- Use the official `@anthropic-ai/sdk` (never raw fetch in a TS project).
- Define the schema once in **Zod** and pass `output_config: { format: zodOutputFormat(RoomProgramSchema) }` to `client.messages.parse(...)`. `parse()` validates + deserializes for you; read `response.parsed_output` (it is null on failure â€” guard it). One Zod source of truth shared by the API route AND the SVG renderer's plan model.
- Model: `claude-opus-4-8`, `thinking: { type: "adaptive" }` (adaptive only â€” `budget_tokens`/`temperature`/`top_p` are removed and 400 on 4.8). Set `output_config.effort` ('high' is a good default; combine with the format key in the same `output_config`).

**Schema rules (grammar-constrained subset):**
- Every object MUST set `additionalProperties: false`. List `required` explicitly.
- Supported: object/array/string/number/integer/boolean/null, enum, const, anyOf/allOf, $ref/$def, string formats (uuid, date, etc.).
- NOT supported: numeric bounds (minimum/maximum/multipleOf), string length (minLength/maxLength), most array constraints (only minItems 0|1), recursion. Enforce "area â‰¥ 0", "adjacency references a real room id", room-count limits, and non-overlap **in code, not in the schema**.
- Complexity caps: â‰¤20 strict tools, â‰¤24 optional params total, â‰¤16 union params â€” keep the room-program schema reasonably flat (`rooms[]` of `{id, name, type(enum), area, dims, adjacency[]}` is well within limits).

**Validation (defence in depth â€” structure â‰  correctness):** grammar constraints guarantee the SHAPE, not the SEMANTICS. The model can still emit schema-valid-but-wrong data. After `parse()`: (a) Zod already enforced types; (b) add business validation the schema can't express â€” referential integrity of adjacency ids, area/dimension sanity, totals, duplicates â€” and return 422 with details on failure rather than trusting the JSON. Handle `stop_reason: "refusal"` (output won't match schema) and `"max_tokens"` (truncated â†’ raise max_tokens or stream) explicitly.

**Streaming:** structured outputs are compatible with streaming. For a potentially long room program prefer streaming to avoid SDK HTTP timeouts: `client.messages.stream({... output_config ...})` then `await stream.finalMessage()`, and parse/validate the final object. Return one JSON response if the client just needs the finished object (stream server-side for timeout safety). Partial JSON mid-stream isn't independently valid â€” validate only the final assembled object.

**First-request note:** a new schema has a one-time compile cost, then is cached ~24h â€” negligible after warmup.
